# Техническое задание (ГОСТ 34.602-2020)

## 1. Общие сведения

**Наименование:** AI-приложение для анализа почтовых сообщений.
**Шифр темы:** ПДП-ВКР-2026-НАГ.
**Исполнитель:** Нефедов Алексей Геннадьевич (МУИВ, бакалавр).
**Руководитель от университета:** Коротков Дмитрий Павлович.
**Срок выполнения:** 27.04.2026 — 24.05.2026.

## 2. Назначение и цели создания системы

### 2.1 Назначение

Система предназначена для автоматического анализа входящих почтовых
сообщений пользователя с целью снижения трудозатрат на рутинную
обработку переписки.

### 2.2 Цели создания

- Автоматически классифицировать письма по категориям.
- Обнаруживать спам и фишинг.
- Сокращать длинные письма до 2–3 предложений.
- Извлекать из писем структурированные данные (даты, суммы, задачи).
- Выполнять поиск по смыслу, а не только по ключевым словам.

## 3. Характеристики объекта автоматизации

Объектом автоматизации является **процесс обработки почтовых сообщений
работника офиса / аналитика / студента**:

- AS-IS: пользователь вручную просматривает 50–200 писем в день,
  вручную сортирует по папкам, вручную копирует задачи в планировщик.
- TO-BE: система автоматически классифицирует, помечает опасные,
  извлекает структурированные элементы и подаёт сводку.

## 4. Требования к системе

### 4.1 Требования к системе в целом

- Архитектура — клиент-серверная.
- Язык реализации — Python 3.11+.
- Серверная часть — FastAPI.
- Клиент — PyQt6-десктоп.
- СУБД — PostgreSQL (prod) или SQLite (dev/test).
- Протокол связи — HTTPS (в prod).

### 4.2 Функциональные требования

| № | Требование | Реализация |
|---|---|---|
| F1 | Регистрация / аутентификация | `/auth/register`, `/auth/login`, JWT |
| F2 | Импорт писем по IMAP | `mail/imap_client.py` + `ImportDialog` |
| F3 | Автоклассификация писем | `ml/classifier.py` |
| F4 | Обнаружение спама и фишинга | `ml/spam_detector.py` |
| F5 | Суммаризация | `ml/summarizer.py` |
| F6 | Извлечение сущностей | `ml/entity_extractor.py` |
| F7 | Пользовательские правила | `db.Rule` + `RulesWindow` |
| F8 | Экспорт отчётов .docx/.xlsx | `utils/reports.py` + `ReportsWindow` |
| F9 | Роли ≥ 3 (admin/analyst/user) | `db.Role` |
| F10 | Панель администратора | `AdminWindow` + `/admin/users` |
| F11 | Личный кабинет | `ProfileWindow` |
| F12 | Встроенная справка | `HelpWindow` |
| F13 | ≥ 10 оконных форм | 12 окон (превышает требование) |
| F14 | Панель настроек с ≥ 5 пунктов | `SettingsWindow` 5 вкладок |

### 4.3 Нефункциональные требования

- Время отклика: ≤ 2 с на одно письмо (без LLM-API).
- Оффлайн-работа: базовая функциональность без интернета (локальный ML).
- Кроссплатформенность: Windows 10/11 (основная), Linux/macOS (через PyInstaller).
- Оригинальный объём кода: ≥ 2 000 логических строк (требование методички).

### 4.4 Требования к надёжности

- Отказ IMAP-сервера не должен приводить к падению приложения
  (отображается MessageBox с ошибкой, окно импорта можно переоткрыть).
- Ошибки анализа не должны приводить к повреждению БД.

### 4.5 Требования к безопасности

- Пароли только в bcrypt-хешах.
- JWT ограничен по времени (24 ч по умолчанию).
- Административные операции — только для роли `admin`.
- Запрещено хранить пароли от IMAP в открытом виде (предусмотрен
  ввод только в диалоге, без сохранения).

## 6. Порядок контроля и приёмки

Проверяется по критериям:

1. Количество коммитов в Git ≥ 50.
2. Оригинальный код ≥ 2 000 строк (без пустых и комментариев).
3. Все тесты `pytest` — зелёные.
4. `.exe` запускается на чистой Windows-машине.
5. Отчёт по ПДП — соответствие структуре методички.
